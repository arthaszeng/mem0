# v2.1.0 重构：数据过滤从 user_id → project_id

## 状态：计划已确认，尚未开始实现

计划文件：`.cursor/plans/project-scoped_data_filtering_902cabac.plan.md`

## 背景

导入他人记忆后，前端 Filter 里出现了小写 `memverse` 和 `openmemory`（已修复数据和导入代码）。
更根本的问题：所有数据过滤用 `user_id`，导致导入的记忆/实体可能因创建者不同而不可见。

## 核心原则

1. **数据范围由项目决定**：`Memory.project_id` / Qdrant `project_id` 是唯一过滤维度
2. **user_id 保留为创建者元数据**：写入时仍记录，但不用于范围过滤
3. **无 project_slug 时回退到默认项目**：用户注册时自动创建同名默认项目
4. **权限通过项目成员身份控制**：不再用 `memory.user_id != user.id`
5. **记忆归属项目不归用户**：导入到项目 C 的记忆就属于项目 C（clone 语义）

## 删除逻辑

- 删除项目：只有 owner + superadmin 可删；default project 不可删（包括 superadmin）
- 删除项目 = 级联删除：project_id → memories + Qdrant vectors + Kuzu entities + categories + logs
- 删除用户 = 删除其 owned projects → 级联 + 清理 memberships/apps

## 改动范围（6 个文件，~64 处）

### Layer 0: 模型 + 删除保护
- `models.py`: Project 加 `is_default = Column(Boolean, default=False)`
- `projects.py`: delete_project 权限收紧到 owner，is_default 拒删
- `gateway_auth.py`: `_ensure_default_project()` 设置 `is_default=True`

### Layer 1: 核心工具函数 (`gateway_auth.py`)
- 新增 `resolve_project_required(auth, db, project_slug)` — 始终返回 ProjectContext
- 新增 `check_memory_project_access(db, memory, auth)` — 检查项目成员身份

### Layer 2: REST API (`memories.py` ~30 处)
- 数据过滤：`Memory.user_id == user.id` → `Memory.project_id == pctx.project_id`
- Qdrant 过滤：`FieldCondition(key="user_id")` → `FieldCondition(key="project_id")`
- 权限检查：`memory.user_id != user.id` → `check_memory_project_access()`
- 实体过滤：`_get_accessible_memory_ids` 始终用 project_id

### Layer 3: MCP Server (`mcp_server.py` ~17 处)
- search_memory, list_memories, delete_all, update, archive/restore, export, insights
- `_resolve_project()` 改为始终返回有效 project_id

### Layer 4: 其他路由 (~16 处)
- `backup.py`: export/reembed/clear
- `mcp_prompts.py`: briefing/search
- `stats.py`: get_profile
- `apps.py`: 统计

### Layer 5: 删除逻辑重构 (`projects.py`)
- delete_project: 权限 owner+superadmin，is_default 保护
- purge_user: 改为遍历 owned projects → 级联，去掉 orphan memories 的 user_id 查询
